# 原型链

本章用于建立 JavaScript 对象系统的底层认知：属性为什么能“沿链查找”、继承为什么本质是原型委托。

## 学习目标

- 理解 `prototype`、`__proto__`、`[[Prototype]]`、构造函数之间的关系
- 掌握属性查找路径与遮蔽（shadowing）机制
- 能解释 `new`、`instanceof` 在原型链上的行为
- 理解 `class` 语法与原型链的对应关系
- 避免常见原型相关误区并形成可维护实践

## 1. 核心概念与术语对齐

### 1.1 什么是原型

- 每个对象都有一个内部原型指针（`[[Prototype]]`）
- `[[Prototype]]` 指向另一个对象，用于属性委托查找

### 1.2 `prototype` vs `__proto__`

- `prototype`：函数对象（尤其构造函数）上的属性
- `__proto__`：对象访问其 `[[Prototype]]` 的历史访问器（不推荐业务代码依赖）
- 推荐使用 `Object.getPrototypeOf` / `Object.setPrototypeOf`

### 1.3 构造函数、实例、原型对象关系

- `Ctor.prototype` 是“实例共享方法”的存放处
- `instance.__proto__ === Ctor.prototype`（概念上成立）
- `Ctor.prototype.constructor` 默认回指 `Ctor`

## 2. 属性查找机制

### 2.1 查找路径

- 先查对象自身属性
- 未命中则沿 `[[Prototype]]` 向上查找
- 直到 `null` 终止

### 2.2 属性遮蔽（Shadowing）

- 实例同名属性会遮蔽原型属性
- 删除实例属性后会“重新显现”原型上的同名属性

### 2.3 读写差异

- 读取：沿链查找
- 赋值：默认写到当前对象（不会改原型对象同名属性）

## 3. `new` 与 `instanceof` 的原型链视角

### 3.1 `new` 的关键步骤

- 创建新对象
- 连接新对象原型到 `Ctor.prototype`
- 调用构造函数并绑定 `this`
- 返回实例对象

### 3.2 `instanceof` 判断原理

- 判断 `Ctor.prototype` 是否出现在实例的原型链上
- 可用来解释“跨 iframe/realm”判断差异

## 4. 继承与原型链

### 4.1 原型链继承本质

- 子对象通过原型委托复用父对象成员
- 共享的是原型上的方法，不是实例私有数据

### 4.2 组合继承与寄生组合继承（概念）

- 为什么早期需要这些模式
- 它们解决了哪些复用与构造调用问题

### 4.3 `Object.create` 的角色

- 显式创建“指定原型”的对象
- 常用于纯对象继承与字典对象创建

## 5. `class` 与原型链的关系

### 5.1 `class` 不是新继承机制

- `class` 是原型链继承的语法糖
- `extends` / `super` 最终仍落在原型链与构造调用上

### 5.2 类成员的归属

- 实例方法：挂在 `ClassName.prototype`
- 静态方法：挂在 `ClassName` 本身
- 私有字段：语法层封装，不参与原型共享

### 5.3 与 `类` 章节联动

- 参考：[类](../语言重难点/类.md)

## 6. 常见误区清单

- 误以为 `prototype` 是每个对象都有的属性
- 误把 `__proto__` 当成推荐 API 到处使用
- 在原型上放可变引用类型数据导致实例互相污染
- 重写 `prototype` 后忘记修复 `constructor`
- 误解“方法定义在构造函数里”和“定义在原型上”的性能差异

## 7. 性能与实践建议

- 方法尽量定义在原型上，避免每次实例化重复创建函数
- 避免频繁动态修改对象原型（影响引擎优化）
- 使用 `Object.create(null)` 创建纯字典对象时，注意无原型方法
- 对外优先暴露稳定接口，弱化对原型细节的直接依赖

## 8. 练习题建议

### 8.1 基础题

- 画出给定对象的原型链
- 解释某个属性最终命中的是哪一层

### 8.2 进阶题

- 手写一个简化版 `new`（只保留核心步骤）
- 手写一个简化版 `instanceof`

### 8.3 综合题

- 将构造函数写法重构为 `class`，并标注每个成员对应到哪里
- 排查一个“实例间数据污染”问题并给出修复方案
