# 模块化

本章聚焦 JavaScript 模块化的核心思想与工程实践，帮助你从“会写 import/export”进阶到“能定位模块问题”。

## 学习目标

- 理解为什么需要模块化，以及它解决了什么问题
- 掌握 CommonJS、ESM、UMD、IIFE 的差异和适用场景
- 熟悉 ESM 的导入导出语法与默认/具名导出的边界
- 能分析运行时与构建时行为差异，识别循环依赖风险
- 了解 Node.js 与浏览器环境在模块化上的关键区别

## 1. 模块化的动机

### 1.1 没有模块化时的问题

- 全局变量污染
- 命名冲突
- 依赖关系混乱
- 文件组织与复用困难

### 1.2 模块化带来的收益

- 封装与隔离
- 依赖显式化
- 可维护性提升
- 更利于团队协作与测试

## 2. 模块化方案演进与对比

### 2.1 IIFE（立即执行函数）

- 通过函数作用域实现“伪模块”
- 适合早期浏览器脚本隔离
- 局限：依赖管理和复用能力弱

### 2.2 CommonJS（CJS）

- 核心语法：`require` / `module.exports`
- 特点：同步加载、运行时加载
- 常见场景：Node.js 传统生态

### 2.3 UMD

- 目标：兼容 AMD、CommonJS、浏览器全局
- 特点：发布库时的“通吃”方案
- 局限：写法复杂、现代项目使用减少

### 2.4 ES Module（ESM）

- 核心语法：`import` / `export`
- 特点：静态结构、可做 Tree Shaking
- 场景：现代前端与 Node.js 新项目主流

### 2.5 速查对比（建议课堂表格展开）

- 语法风格
- 加载时机（静态/动态）
- 是否支持静态分析
- 环境兼容性
- 工具链友好度

## 3. ESM 核心语法

### 3.1 导出：`export`

- 具名导出（Named Export）
- 默认导出（Default Export）
- 统一导出（聚合导出）

### 3.2 导入：`import`

- 按名称导入
- 默认导入
- 别名导入（`as`）
- 动态导入（`import()`）

### 3.3 默认导出 vs 具名导出

- 何时用默认导出
- 何时优先具名导出
- 混用时的可读性与维护成本

## 4. 运行时与构建时差异

### 4.1 运行时视角

- 模块何时执行
- 模块缓存机制
- 副作用代码的触发时机

### 4.2 构建时视角

- 静态分析与依赖图
- Tree Shaking 的前提条件
- 打包器对 CJS/ESM 的处理差异

## 5. 循环依赖

### 5.1 什么是循环依赖

- A 依赖 B，同时 B 依赖 A
- 直接循环与间接循环

### 5.2 在不同模块系统中的表现

- CommonJS 的导出对象“部分可用”问题
- ESM 的 live binding 与初始化时序

### 5.3 排查与规避策略

- 拆分公共模块
- 延迟依赖（按需引入）
- 降低跨层耦合

## 6. Node.js 与浏览器差异

### 6.1 Node.js 中的模块化

- `.cjs` / `.mjs` 与 `package.json` 的 `type`
- CJS 与 ESM 互操作边界
- `__dirname`、`import.meta.url` 等差异点

### 6.2 浏览器中的模块化

- `<script type="module">`
- 模块按 URL 加载
- 跨域与路径解析注意事项

## 7. 工程实践建议

- 团队统一导出风格（默认/具名规则）
- 目录与入口约定（`index` 聚合出口）
- 控制模块职责，避免“超级工具模块”
- 尽量减少副作用模块，必要时显式标注
- 为公共模块补充类型与文档说明

## 8. 常见坑清单

- `import` 路径大小写在不同系统表现不一致
- 默认导出与具名导入写反
- 混用 CJS/ESM 导致导入结果不符合预期
- 循环依赖引发 `undefined` 或初始化错误
- 误以为“能运行就能 Tree Shaking”

## 9. 练习题建议

### 9.1 基础题

- 将一个脚本拆分为 3 个 ESM 模块并正确导入
- 分别实现默认导出与具名导出，并互相改写

### 9.2 进阶题

- 将 CommonJS 项目中的核心模块迁移到 ESM
- 设计一个 `index` 聚合导出层，优化导入路径

### 9.3 综合排错题

- 给定一个循环依赖示例，定位问题并重构
- 给定一个 CJS/ESM 混用报错案例，修复并说明原因
