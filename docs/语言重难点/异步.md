# 异步

本章聚焦 JavaScript 异步编程的核心机制与工程实践，目标是让你不仅会写，还能解释“为什么这样执行”。

## 学习目标

- 区分同步/异步、阻塞/非阻塞的概念边界
- 理解事件循环中调用栈、任务队列、微任务与宏任务的执行顺序
- 掌握回调、Promise、`async/await` 的演进关系与适用场景
- 能设计常见并发策略，并处理超时、取消、错误传播
- 能定位异步代码中的高频问题并进行调试

## 1. 基础概念

### 1.1 同步与异步

- 同步：按顺序执行，前一步完成后才进入下一步
- 异步：发起任务后不阻塞主流程，结果稍后通知

### 1.2 阻塞与非阻塞

- 阻塞：当前线程必须等待操作完成
- 非阻塞：当前线程可继续处理其他任务
- 注意：同步/异步 与 阻塞/非阻塞 不是一一对应关系

### 1.3 为什么需要异步

- I/O（网络、文件、数据库）等待时间长
- 提升吞吐与响应能力
- 避免 UI 卡顿与主线程长时间占用

## 2. 事件循环（Event Loop）

### 2.1 关键组成

- 调用栈（Call Stack）
- 宏任务队列（Macro Task Queue）
- 微任务队列（Micro Task Queue）

### 2.2 执行流程主线

- 先执行同步代码
- 当前宏任务结束后，清空所有微任务
- 再进入下一个宏任务

### 2.3 常见任务类型

- 宏任务：`setTimeout`、`setInterval`、I/O 回调
- 微任务：`Promise.then/catch/finally`、`queueMicrotask`

### 2.4 课堂重点

- 为什么 `Promise.then` 往往先于 `setTimeout` 执行
- “一次事件循环”里微任务会被批量清空

## 3. 回调与回调地狱

### 3.1 回调函数模式

- 通过函数参数接收异步结果
- 错误优先回调（Node 风格）

### 3.2 回调地狱问题

- 嵌套层级深
- 错误处理分散
- 可读性与维护性差

### 3.3 改造思路

- 拆分命名函数
- 封装 Promise
- 使用 `async/await` 降低嵌套

## 4. Promise 核心

### 4.1 状态与不可逆

- `pending`、`fulfilled`、`rejected`
- 一旦状态确定不可再次改变

### 4.2 链式调用

- `then` 返回新 Promise
- 值透传与 Promise 展开（flatten）

### 4.3 错误处理

- `catch` 捕获链路中的异常
- `finally` 用于收尾，不改变最终值（通常）

### 4.4 Promise/A+ 关键条款（最小集）

- Promise 只有三态：`pending`、`fulfilled`、`rejected`，且状态一旦变更不可逆
- `then(onFulfilled, onRejected)` 必须返回一个新的 Promise（记为 `promise2`）
- `onFulfilled` 仅在 fulfilled 时调用，`onRejected` 仅在 rejected 时调用
- 同一个 Promise 可以多次调用 `then`；对应回调按注册顺序执行
- 回调必须异步触发（当前执行栈清空后再执行）
- 若处理器不是函数：
  - `onFulfilled` 非函数时，值应“透传”到下一个 `then`
  - `onRejected` 非函数时，拒因应继续向后抛出
- 回调抛出异常时，`promise2` 必须变为 rejected，拒因为该异常

### 4.5 Promise Resolution Procedure（`[[Resolve]]`）

当 `then` 回调返回 `x` 时，`promise2` 的状态由解析过程决定：

- 若 `promise2 === x`，必须以 `TypeError` 拒绝（防止循环引用）
- 若 `x` 是 Promise，则采用其最终状态
- 若 `x` 是对象/函数且含 `then`，按 thenable 处理（同化）
- 取 `then` 或执行 `then` 过程中若抛错，`promise2` 需 rejected
- 对 thenable 的 `resolve/reject` 调用只允许首次生效（防止多次决议）
- 若 `x` 为普通值，`promise2` 直接 fulfilled，值为 `x`

### 4.6 Promise 设计建议

- 每条异步链路保留统一错误出口
- 避免在 `then` 中混入过多业务分支
- 封装 Promise 工具函数时，优先遵循 A+ 解析语义

### 4.7 Promise/A+ 与 ES Promise 的边界

- Promise/A+ 重点规范的是 `then` 行为与解析过程（`[[Resolve]]`）
- `Promise.resolve/reject/all/allSettled/any/race/finally` 属于 ES 规范层面的 API 扩展
- 微任务调度细节由宿主与 ECMAScript 规范共同约束，不完全由 A+ 定义

## 5. `async/await`

### 5.1 语法本质

- `async` 函数返回 Promise
- `await` 暂停当前 `async` 函数，不阻塞主线程

### 5.2 常见写法

- 串行：多个 `await` 按顺序等待
- 并行：先创建 Promise，再统一 `await`

### 5.3 错误处理模式

- 局部 `try/catch`
- 函数级统一捕获
- 与全局错误监控配合

## 6. 并发控制

### 6.1 `Promise.all`

- 全成功才成功
- 任一失败立即失败（fail-fast）

### 6.2 `Promise.allSettled`

- 等待全部结束
- 适合“部分失败可接受”场景

### 6.3 `Promise.race`

- 最先 settle 的结果生效
- 适合超时兜底与竞速策略

### 6.4 `Promise.any`

- 任一成功即成功
- 全失败时抛 `AggregateError`

### 6.5 并发数限制（扩展）

- 大批量请求需限流（如分批/队列）
- 防止服务压垮或浏览器资源耗尽

## 7. 取消与超时

### 7.1 取消机制

- `AbortController` / `AbortSignal`
- 与 `fetch` 等 API 配合

### 7.2 超时处理

- `Promise.race` + 超时 Promise
- 超时后记得中断或清理资源

### 7.3 资源清理

- 取消定时器
- 释放监听器
- 避免内存泄漏

## 8. 常见陷阱与调试建议

### 8.1 常见陷阱

- 忘记 `await` 导致拿到 Promise 而非结果
- `forEach` + `async` 误以为会按顺序等待
- Promise 链中漏 `return` 导致链路断裂
- 只写成功分支，不处理拒绝分支
- 并发场景下共享可变状态造成竞态条件

### 8.2 调试建议

- 用时间戳日志标记任务开始/结束
- 区分业务错误与系统错误
- 在关键链路加统一 `catch` 与兜底告警
- 使用浏览器 DevTools / Node 调试器观察异步调用栈

## 9. 练习题建议

### 9.1 基础题

- 判断多段代码输出顺序（同步/微任务/宏任务）
- 将回调风格函数改写为 Promise

### 9.2 进阶题

- 使用 `async/await` 重构 Promise 链
- 分别用 `all`、`allSettled` 完成批量请求任务并比较行为
- 手写一个最小版 `then` 链，验证“值透传、异常穿透、thenable 同化”

### 9.3 综合题

- 实现“带超时与取消”的请求封装
- 设计一个并发限制器（如最多 3 个任务同时执行）
- 给定一段存在竞态条件的代码，定位并修复
